Firebase Rules Stage 3 콘솔 적용 기록
=====================================

현재 상태
---------
Stage 3에서 준비한 `database.rules.console-candidate.json`은 Stage 4에서 Firebase Console에 적용되었고, GM/플레이어 2계정 테스트를 통과했다.

사용 파일
---------
- 현재 운영 기준: `database.rules.json`
- 적용 후보 기록: `database.rules.console-candidate.json`
- 검토용 초안: `database.rules.hardening-draft.json`
- 롤백용 기존 Rules: `database.rules.rollback-current.json`

적용된 핵심 변경점
-----------------
1. 방 전체 삭제
   - owner 또는 GM만 `rooms/{roomCode}` 전체 삭제 가능.
   - 일반 플레이어가 마지막으로 방을 나가도 방 전체 삭제 불가.

2. players 권한 상승 방지
   - 상위 players 전체 write를 열지 않는다.
   - 플레이어 본인이 role 또는 permissions를 직접 상승시키는 것은 금지.
   - owner/GM은 role/permissions 관리 가능.

3. BGM / 맵세팅 데이터 분리
   - `bgm/playlist`, `bgm/currentTrack`: owner/GM 또는 manageBgm.
   - `bgm/mapBackground`, `mapObjects`, `mapLayerState` 등 맵세팅 키: owner/GM 또는 manageMap.
   - 그 외 bgm 하위 키: owner/GM만.
   - 현재 BGM 기능은 미구축이라 manageBgm 실제 테스트는 보류.

4. Tokens
   - owner/GM은 전체 가능.
   - createToken: 새 토큰 생성.
   - moveToken: x/y 좌표 이동.
   - editToken: 토큰 편집/삭제.

5. Journals / Handouts
   - 현재 코드는 전체 read 후 프론트 필터링 구조다.
   - 이번 단계에서는 read를 엄격하게 막지 않는다.
   - write는 기존 코드 기준에 맞춰 제한한다.

결과
----
- Firebase Console 적용 완료.
- GM/플레이어 2계정 Stage 4 테스트 통과.
- 운영 기준 Rules는 `database.rules.json`으로 승격됨.

후속 작업
---------
- 권한 UI가 없거나 부족한 기능은 클라이언트 UX를 별도로 정리한다.
- journals/handouts per-item read 하드닝은 별도 로드맵으로 진행한다.
