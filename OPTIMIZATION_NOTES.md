# 최적화 1차 적용 메모

이번 수정은 기능을 크게 바꾸지 않으면서, 현재 구조에서 비용이 큰 부분을 먼저 줄이는 방향으로 진행했습니다.

## 이번에 실제로 손본 부분

- `setupFirebaseListeners()` 재호출 시 기존 Firebase 리스너를 정리하도록 변경
  - 중복 리스너 누적 방지
  - 같은 방 재입장/세션 복구/화면 전환 시 불필요한 네트워크 사용 감소
- 프로필 아바타 동기화 방식을 폴링(`setInterval`) 기반에서 이벤트/저장 시점 기반으로 변경
  - 2초마다 로컬스토리지를 읽고 RTDB에 쓰던 방식 제거
  - 불필요한 쓰기 및 브라우저 부하 감소
- 방 참가/생성 시 플레이어 데이터에 아바타/닉네임 색/잡담 닉네임을 함께 저장
  - 방 안에서 추가 조회/동기화 의존도 감소
- 로그인 시 유저 프로필에서 이름/아바타를 로컬로 복원
  - 로컬 상태와 Firebase 상태 불일치 완화
- 로비의 내 캐릭터 리스너 중복 구독 방지

## 아직 남아 있는 구조적 한계

현재 구조가 **100~300명이 동시에 같은 방에서 쓰는 구조**에는 적합하지 않습니다.
그 이유는 다음과 같습니다.

1. 채팅/잡담이 RTDB `onValue()`로 전체 노드를 다시 읽는 구조
2. 이미지와 아바타가 Base64 문자열이라 데이터 전송량이 큼
3. 전역 상태(`St`, `window.*`) 의존도가 높아 모듈 경계가 약함
4. DOM 렌더링이 문자열 기반 `innerHTML` 중심이라 부분 업데이트 최적화가 제한적임

## 다음 단계 권장

2차 최적화에서는 아래 순서가 가장 효과가 큽니다.

1. 채팅/잡담을 `onChildAdded / onChildChanged / onChildRemoved` 기반으로 변경
2. 이미지/아바타를 Firebase Storage + URL 참조 방식으로 전환
3. 방 메타 / 플레이어 상태 / 채팅을 분리하고, 오래된 채팅은 페이지네이션 적용
4. 공통 렌더 함수와 Firebase 접근 래퍼를 분리해 모듈 중복 제거
